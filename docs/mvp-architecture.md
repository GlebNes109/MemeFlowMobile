# MemeFlow MVP: архитектура и границы первой версии

## Что выбрать

Для этого проекта я бы выбрал **клиент-серверную архитектуру с модульным монолитом на бэкенде** и **feature-first архитектурой на Android**.

Это лучший компромисс для MVP:

- не нужно тащить микросервисы, очереди и распределенные транзакции слишком рано;
- можно быстро закрыть обязательный функционал: аккаунты, мемы, подборки, группы, приватность, базовую модерацию;
- при росте проекта монолит можно безопасно дробить по уже выделенным доменам.

## Почему не стоит усложнять на старте

Главная сложность проекта не в трафике, а в бизнес-логике:

- у одного пользователя может быть много групп;
- у мемов и подборок есть разные режимы доступа;
- контент бывает разного типа: загруженная картинка, внешняя картинка, короткое видео по ссылке;
- нужен минимум модерации до попадания в общий фид.

Из-за этого важнее сделать чистую доменную модель и нормальный контракт, чем сразу строить “большую распределенную систему”.

## Рекомендуемый контур системы

### Mobile

- Android-приложение на `Jetpack Compose`
- `Single-Activity + Navigation Compose`
- `MVVM + StateFlow`
- `Repository`-слой между UI и API
- сетевой клиент через `Retrofit` или `Ktor Client`
- DTO и модели API лучше генерировать из OpenAPI, чтобы мобильный клиент не расходился с бэкендом

### Backend

- один backend-приложение c REST API
- отдельные модули внутри монолита:
  - `auth`
  - `users`
  - `groups`
  - `media`
  - `memes`
  - `collections`
  - `feed`
  - `moderation`
- `PostgreSQL` для основных данных
- `S3-compatible object storage` для картинок и превью
- фоновый worker для:
  - импорта внешних ссылок
  - извлечения preview/metadata
  - базовой модерации

### Что важно зафиксировать сразу

- **не хранить `group_id` в `User`**: связь должна быть many-to-many через membership-таблицу;
- **не хранить `collection_id` в `Meme`**: лучше many-to-many через `collection_items`;
- права доступа хранить не “как получится”, а явно:
  - у мема отдельно хранить **direct access** и **effective access**
  - `visibility = public | private | groups`
  - `shared_group_ids[]`
  - UI-термин “ограниченный доступ” в API соответствует значению `groups`

## Модель доступа и правила пересчета

### Уровни доступа

И у мемов, и у подборок есть три уровня доступа:

- `private`: не видит никто, кроме автора и модератора;
- `groups`: видят только участники указанных групп;
- `public`: видят все.

### Почему у мема два типа доступа

Для мема нужно хранить два состояния:

- **direct access**: доступ, который автор задал самому мему вручную;
- **effective access**: итоговый доступ, который получается после учета всех подборок, куда этот мем входит.

Это нужно из-за правила продукта: подборка может “поднять” доступ мемов внутри себя.

### Правило вычисления effective access

Итоговый доступ мема считается как максимум среди:

- его собственного `direct access`;
- доступов всех подборок, в которые этот мем включен.

Приоритет уровней такой:

- `public` сильнее `groups`
- `groups` сильнее `private`

### Что происходит при разных комбинациях

- если мем `private`, а подборка `groups`, мем получает `effective_visibility = groups`, а `effective_shared_group_ids` включает группы подборки;
- если мем `private` или `groups`, а подборка `public`, мем получает `effective_visibility = public`;
- если мем `public`, а подборка `private`, мем остается `public`;
- если мем состоит в нескольких `groups` подборках, итоговый список групп у мема — это объединение всех групп из direct и collection access;
- если мем состоит хотя бы в одной `public` подборке, его `effective_visibility = public`.

### Что важно технически

Подборка **не должна напрямую затирать direct access мема**. Иначе потом нельзя будет корректно откатить доступ при удалении мема из подборки или при понижении доступа подборки.

Поэтому на бэкенде нужно:

- хранить `direct_visibility` и `direct_shared_group_ids` как пользовательский источник истины;
- отдельно хранить или вычислять `effective_visibility` и `effective_shared_group_ids`;
- пересчитывать `effective access` при каждом изменении связей “мем ↔ подборка” и при каждом изменении доступа у мема или подборки.

### Когда обязателен пересчет effective access

- при создании мема;
- при изменении direct access у мема;
- при создании подборки;
- при изменении доступа подборки;
- при добавлении мема в подборку;
- при удалении мема из подборки;
- при удалении подборки.

### Как это выглядит для пользователя

Для пользователя правило должно звучать так:

- мем имеет свой собственный доступ;
- если мем попадает в более открытую подборку, мем становится доступен не уже, чем эта подборка;
- если подборка потом станет более закрытой или мем удалят из подборки, итоговый доступ мема пересчитается заново по оставшимся источникам доступа.

То есть на экране редактирования мема полезно показывать не только “доступ мема”, но и пояснение:

- какой direct access задал автор;
- какой effective access получился в итоге;
- какие подборки расширяют доступ к этому мему.

## Как я бы ограничил MVP

### Что точно делаем

- регистрация и логин
- публичный фид мемов
- профиль пользователя
- загрузка картинки файлом
- сохранение картинки по URL
- сохранение `YouTube Shorts` как внешнего видео-объекта
- создание подборок
- группы и инвайты в группы
- приватность через `public / private / groups`
- репорты и модерация `pending/approved/rejected`

### Что лучше отложить

- “умная” релевантность фида
- прямой шаринг в отдельных пользователей
- сложные ACL на уровне каждого участника
- скачивание и хостинг чужих short-video к себе
- TikTok/Instagram import до отдельной юридической и технической проверки

## Отдельно про шортсы

Самое мутное место в исходном описании — “возможность сохранения ют шортс”.

Для MVP я бы трактовал это так:

- пользователь вставляет ссылку на `YouTube Shorts`;
- бэкенд валидирует URL;
- сохраняет метаданные, canonical URL и thumbnail;
- в приложении этот объект отображается как мем с типом `external_video`.

То есть в первой версии мы **сохраняем short в ленте и подборках как сущность платформы, а не скачиваем видеофайл себе на сервер**. Это зафиксированное решение для MVP.

## Как строить Android-приложение

Поскольку сейчас проект маленький, я бы не дробил его сразу на 10 Gradle-модулей. Для MVP достаточно одного `app`-модуля, но с нормальной структурой пакетов:

```text
com.memeflow.app
  core/
    network/
    model/
    ui/
  feature/
    auth/
    feed/
    upload/
    profile/
    groups/
    collections/
  data/
    remote/
    repository/
```

Внутри фич:

- `ui/`
- `viewmodel/`
- `contract/` или `state/`
- `mapper/` при необходимости

## Как строить backend

На бэкенде я бы держал границы модулей такими:

- `auth`: регистрация, логин, refresh-token
- `users`: профиль, аватар, публичная страница
- `groups`: группы, участники, инвайты
- `media`: прием картинок и импорт внешних URL
- `memes`: создание/обновление/удаление мемов
- `collections`: подборки и элементы подборок
- `feed`: выдача общедоступного или доступного контента
- `moderation`: статусы, репорты, очередь на разбор

Если позже проект вырастет, первыми кандидатами на вынос в отдельные сервисы будут `media` и `moderation`.

## Зафиксированные продуктовые решения

1. `YouTube Shorts` в MVP сохраняется как внешняя ссылка с метаданными и preview, без загрузки видеофайла на наш сервер.
2. Один мем может входить в несколько подборок.
3. Подборка может расширять доступ к мемам внутри себя, но не сужает его. Итоговый доступ мема определяется через `effective access`.
4. В группах пока только две роли: `owner` и `member`.
5. Глобальный поиск ищет пользователей, подборки, теги мемов и подписи мемов.

## Что это значит для mobile API

Чтобы мобильное приложение можно было писать без догадок, API должно возвращать:

- для мема: и `direct`, и `effective` доступ;
- для подборки: ее собственный доступ и список мемов в порядке отображения;
- для экрана “мой профиль”: не только публичные, но и все мои мемы/подборки;
- для экрана редактирования мема: список подборок, которые расширяют ему доступ.
